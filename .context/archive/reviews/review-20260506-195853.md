## Findings

### Finding 1
- Severity: medium
- Category: bug
- Summary: Modifying the existing `features` table inside `CREATE TABLE IF NOT EXISTS` does not alter tables that already exist. Existing databases will lack the new `date_source` column.
- Evidence: `internal/db/db.go` `Migrate()` splits `schema.sql` and runs each statement verbatim. On an existing DB, `CREATE TABLE IF NOT EXISTS features(...)` is a no-op, so `date_source` is never added. `go test ./...` passes only because every test uses a fresh database.
- Recommendation: Add conditional migration logic in `db.go` (or standalone ALTER TABLE guards) to apply column additions to existing tables. Alternatively, document that existing DBs must be rebuilt until a migration framework is adopted.
- Rule candidate: Schema changes that add columns to existing tables require explicit migration handling beyond `CREATE TABLE IF NOT EXISTS`.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: `internal/db/db_test.go` was not updated to assert the new `stories` table and its indexes exist after `Open`.
- Evidence: `TestOpenCreatesSchema` checks `epics`, `features`, `sprints`, and `date_audit_logs` tables, plus a fixed set of indexes. It omits `stories`, `idx_stories_feature_id`, and `idx_stories_sprint` even though `schema.sql` now defines them.
- Recommendation: Add `assertTableExists(t, database, "stories")`, `assertIndexExists(t, database, "idx_stories_feature_id")`, and `assertIndexExists(t, database, "idx_stories_sprint")` to `TestOpenCreatesSchema`.
- Rule candidate: Any change to `schema.sql` must be accompanied by a matching update to the schema smoke test (`db_test.go` or equivalent) so that missing tables/indexes are caught in CI.

### Finding 3
- Severity: low
- Category: test-gap
- Summary: No negative-path test verifies that the `stories` foreign key constraint rejects inserts with a non-existent `feature_id`.
- Evidence: `TestStoryCRUD` and `TestStoryDateSource_RoundTrip` both seed a valid feature before creating stories. There is no `TestStoryCreate_InvalidFeatureID` or similar test.
- Recommendation: Add a test that attempts to create a story referencing `feature_id = "F-NOT-FOUND"` and asserts the insert fails.
- Rule candidate: Every new foreign key should have a negative-path test verifying constraint enforcement.

### Finding 4
- Severity: low
- Category: simplification
- Summary: The `stories` table definition is inserted between existing `CREATE INDEX` statements, making `schema.sql` harder to scan.
- Evidence: In `schema.sql`, `CREATE TABLE stories` appears after `idx_date_audit_logs_entity` and before `idx_stories_feature_id`, breaking the natural grouping of tables-followed-by-indexes.
- Recommendation: Reorder `schema.sql` so all `CREATE TABLE` statements appear first, followed by all `CREATE INDEX` statements.
- Rule candidate: Keep schema DDL grouped by type (tables first, then indexes, then views, etc.).
