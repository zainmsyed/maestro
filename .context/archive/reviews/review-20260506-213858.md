## Findings

### Finding 1
- Severity: medium
- Category: test-gap
- Summary: `ParseAssignedTo` handles three distinct formats (angle-bracket email, parenthesis email, plain email) observed in real exports, but has zero direct unit tests.
- Evidence: `internal/importer/hierarchy.go` `ParseAssignedTo` contains branching logic for `"First Last <email>"`, `"Last, First <email>"`, `"Name (email)"`, and plain email, yet `importer_test.go` only exercises it indirectly through the integration test. A regression in any branch would only be caught by the full fixture import.
- Recommendation: Add `TestParseAssignedTo` covering at least: angle-bracket with comma name, parenthesis wrapper, plain email only, and empty string.
- Rule candidate: Every parser helper with ≥2 branching formats observed in real data should have direct unit tests.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: No test verifies the fallback behavior when a row's `Parent` ID points to an entity of the wrong type (e.g., a story parented to an epic).
- Evidence: `createFeature` checks `parent.itemType == "epic"`; `createStory` checks `parent.itemType == "feature"`. If the type check fails, the entity is routed to the synthetic unassigned bucket. This path is exercised by real data but not by any explicit test case.
- Recommendation: Add an integration test fixture row where a story's `Parent` is an epic ID, and assert it becomes an orphan assigned to the synthetic unassigned feature.
- Rule candidate: Hierarchy linkers must have explicit negative-path tests for wrong-type parent references.

### Finding 3
- Severity: low
- Category: simplification
- Summary: `rawEntity` carries four fields (`dateFormat`, `ambiguous`, `rawDate`, `areaPath`) that are populated during Pass 1 but never read during Pass 2.
- Evidence: In `internal/importer/types.go`, `rawEntity` declares these fields. `dateFormat` and `ambiguous` are consumed during Pass 1 to populate the import report. `rawDate` is used for `AmbiguousDateCandidate.RawDate` in Pass 1. `areaPath` is never read anywhere. After Pass 1, only `id`, `parentID`, `itemType`, `title`, `state`, `owner`, `sprint`, `storyPoints`, and `targetDate` are read in Pass 2.
- Recommendation: Remove the four unused fields from `rawEntity` and compute/report ambiguous-date data inline during Pass 1 without storing it on the struct. Drop `areaPath` entirely unless a future story needs it.
- Rule candidate: Struct fields that are written in one pass and never read in subsequent passes should be eliminated or inlined.

### Finding 4
- Severity: low
- Category: test-gap
- Summary: Synthetic ID generation for empty IDs is tested only for stories; empty epic and empty feature IDs are not covered.
- Evidence: `TestImportCSV` asserts `len(report.SyntheticStoryIDs) == 2` but the fixture also contains an empty epic ID and an empty feature ID. The code generates `epic-auto-*` and `feature-auto-*` for them, but no test verifies their creation or database presence.
- Recommendation: Extend `TestImportCSV` to assert that the empty-epic and empty-feature rows received synthetic IDs and were persisted.
- Rule candidate: Synthetic/ fallback ID generation should be tested for every entity type that can trigger it, not just the most common one.
