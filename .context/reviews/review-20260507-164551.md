## Findings

### 1. Onboarding.svelte is implemented but unmounted
- **Severity:** info
- **Category:** workflow
- **Summary:** `frontend/src/screens/Onboarding.svelte` was created within story-006 scope and is functionally complete, but it is not imported or rendered by `App.svelte`. The temporary visual verification path (`/date_screen`) wires `DateAssignment` directly, bypassing `Onboarding`. This means the routing logic (loading → date-assignment → confirm → complete) cannot be exercised in the running app until story-007 mounts it.
- **Evidence:** `frontend/src/screens/Onboarding.svelte` exists; `frontend/src/App.svelte` imports `DateAssignment` directly for the `/date_screen` temporary route and does not import `Onboarding`.
- **Recommendation:** This is acceptable for story-006 because `Onboarding.svelte` mounting is outside scope. Ensure story-007 mounts `Onboarding` as the entry point after import so the full flow becomes testable.
- **Rule candidate:** —

### 2. ApiError class exported but never imported
- **Severity:** info
- **Category:** dead-code
- **Summary:** `ApiError` is exported from `api.ts` but no consumer imports it. The class is only used internally inside `apiFetch` to throw. Consumers catching errors currently use `instanceof Error` rather than `instanceof ApiError`, so the export is dead surface area.
- **Evidence:** `frontend/src/lib/api.ts` line 8: `export class ApiError extends Error { ... }`. No file imports `ApiError`.
- **Recommendation:** Either remove the `export` keyword to keep it module-private, or update a consumer to use `instanceof ApiError` and read `.status`. Keeping it private is the simpler fix.
- **Rule candidate:** —

### 3. DatePatchResponse type exported but unused
- **Severity:** info
- **Category:** dead-code
- **Summary:** `DatePatchResponse` is declared and exported in `api.ts`, but no component or store imports it. The `patchDate` method returns it via type parameter but callers do not type the result.
- **Evidence:** `frontend/src/lib/api.ts` lines 85–89: `export type DatePatchResponse = { ... }`.
- **Recommendation:** Remove the `export` keyword or delete the type entirely; the return type can be inferred or inlined in `patchDate`.
- **Rule candidate:** —

### 4. assignRows function exceeds complexity threshold
- **Severity:** low
- **Category:** simplification
- **Summary:** `assignRows` in `DateAssignment.svelte` mixes validation, batch state updates, per-row async PATCH calls, success/error tracking, and dispatch logic. The function is ~45 lines and does four distinct things sequentially.
- **Evidence:** `frontend/src/screens/DateAssignment.svelte` lines 88–126: `async function assignRows(...)`.
- **Recommendation:** Extract helpers:
  - `validateAll(targetRows)` → returns `{ valid: boolean; rowsWithErrors: AssignmentRow[] }`
  - `patchRow(row)` → returns `{ success: boolean; message?: string }`
  - Keep `assignRows` as an orchestrator that calls the helpers.
- **Rule candidate:** Functions that mix validation, async I/O, and UI state updates should be split into an orchestrator + helpers.

- **Evidence:** `frontend/src/App.svelte` lines 5–27: `showDateScreen` constant and `mockReport` object.
- **Recommendation:** Keep the temporary mount only until story-007 replaces it with real onboarding wiring. Add a `// TODO(story-007): remove /date_screen temporary test mount` comment.
- **Rule candidate:** —
