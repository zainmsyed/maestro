## Findings

### Finding 1
- **Severity:** low
- **Category:** dead-code
- **Summary:** `frontend/src/lib/listView.ts` exports several helpers that are only consumed within the same module.
- **Evidence:** Verified Fallow findings at lines 34 (`normalizeDate`), 38 (`deriveHealth`), 140 (`normalizeOptionLabel`), 174 (`rowSortValue`), and 188 (`sortRows`). Manual inspection confirms each is referenced only by other functions inside `listView.ts`.
- **Recommendation:** Remove the `export` keyword from `normalizeDate`, `deriveHealth`, `normalizeOptionLabel`, `rowSortValue`, and `sortRows` to narrow the public surface.
- **Rule candidate:** A module should not export symbols that have no external consumers. Re-export only when cross-module consumption is proven.

### Finding 2
- **Severity:** low
- **Category:** simplification
- **Summary:** `FilterBar.svelte` and `GroupBySelect.svelte` define local copies of `GroupBy` and `FilterState` instead of importing the canonical types from `listView.ts`.
- **Evidence:** `FilterBar.svelte` declares `type GroupBy = 'epic' | 'sprint' | 'owner'` and `type FilterState = { epic: string; owner: string; sprint: string; status: string }`. `GroupBySelect.svelte` also declares `type GroupBy`. Both duplicate exports already present in `frontend/src/lib/listView.ts`.
- **Recommendation:** Import `GroupBy` and `FilterState` from `../lib/listView` in both components and delete the local type declarations.
- **Rule candidate:** Svelte components must import shared view-model types from a single source of truth rather than re-declaring them locally.

### Finding 3
- **Severity:** low
- **Category:** simplification
- **Summary:** `filterRows` in `listView.ts` exceeds the complexity threshold with a score of 10 due to four inline filter conditions.
- **Evidence:** `frontend/src/lib/listView.ts:179` ŌĆö the arrow function chains four conditional returns.
- **Recommendation:** Extract a `matchesFilter(row, key, value)` predicate helper and iterate over the active filter keys, or decompose into smaller per-key predicate functions. This makes unit testing individual filter rules trivial.
- **Rule candidate:** Filter functions with more than three hard-coded conditions should be decomposed into predicate helpers.

### Finding 4
- **Severity:** low
- **Category:** simplification
- **Summary:** `optionsForKey` in `ListView.svelte` exceeds the complexity threshold with a score of 6 due to an if-else chain per filter key.
- **Evidence:** `frontend/src/screens/ListView.svelte:89` ŌĆö four branches select a different source field and assemble options.
- **Recommendation:** Replace the if-else chain with a keyed configuration map (e.g., `const fieldByKey = { epic: 'epicTitle', owner: 'owner', sprint: 'sprint', status: 'status' }`) and a single generic extraction path.
- **Rule candidate:** Component helper functions with per-key branching should prefer lookup tables over if-else chains.

### Finding 7
- **Severity:** info
- **Category:** workflow
- **Summary:** Fallow flags `svelte-check` as an unused dev dependency.
- **Evidence:** `frontend/package.json` lists `svelte-check` but the `scripts` block has no `check` entry. It is still used manually during development (`npx svelte-check`).

- **Rule candidate:** Dev dependencies listed in `package.json` should be referenced by at least one npm script or build step.

### Finding 8
- **Severity:** low
- **Category:** test-gap
- **Summary:** `frontend/src/lib/csvExport.ts` has no automated coverage for its edge-case handling.
- **Evidence:** `escapeCsv` handles quotes, commas, and newlines, but no test file covers it. `downloadCsv` manipulates the DOM and is harder to unit-test, but `buildCsv` and `escapeCsv` are pure and testable.
- **Recommendation:** Add a lightweight `csvExport.test.ts` covering `escapeCsv` with commas, quotes, newlines, null/undefined values, and `buildCsv` with empty row arrays.
- **Rule candidate:** Any utility that performs string escaping or format conversion must have at least one unit test exercising its edge cases.

### Finding 9

- **Rule candidate:** Components with visual state branches (loading, empty, error) must have test coverage for each branch.
