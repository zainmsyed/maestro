## Findings

### 1. GET requests to /date and /feature suffix paths are routed to PATCH handlers
- **Severity:** medium
- **Category:** bug
- **Summary:** `handleStory` and `handleFeature` dispatch to patch-specific handlers (`patchStoryDate`, `patchFeatureDate`, etc.) based only on URL path suffix, without verifying the HTTP method. A GET request to `/api/stories/S-1/date` is therefore routed to `patchStoryDate`, which attempts to decode an empty body and returns 400 Bad Request instead of 405 Method Not Allowed (or 404).
- **Evidence:** `internal/api/routes.go` lines 42–76. `handleStory` checks `strings.HasSuffix(r.URL.Path, "/date")` and unconditionally calls `s.patchStoryDate(w, r)`.
- **Recommendation:** Add method guards inside dispatch handlers, or register more specific route patterns (e.g., `GET /api/stories/{id}` separately from `PATCH /api/stories/{id}/date`) so the mux never cross-routes methods.
- **Rule candidate:** Route dispatchers must validate HTTP method before delegating to operation-specific handlers.

### 2. SQL table names interpolated via fmt.Sprintf in metrics queries
- **Severity:** medium
- **Category:** bug
- **Summary:** `deadlineHitRate` and `scopeCreepRate` in `repository/metrics.go` build queries with `fmt.Sprintf` and a `%s` table-name parameter. While currently called only with hardcoded strings, this pattern is brittle and dangerous if the parameter ever becomes dynamic.
- **Evidence:** `internal/repository/metrics.go` lines 76–77 and 92–93: `query := fmt.Sprintf("... FROM %s WHERE ...", table)`.
- **Recommendation:** Maintain an explicit allow-list of table names, or write separate query functions per table so identifiers are never injected dynamically.
- **Rule candidate:** Never interpolate SQL identifiers (tables, columns) via fmt.Sprintf or string concatenation; use an explicit whitelist if dynamic identifiers are required.

### 3. N+1 query problem in epic list and get endpoints
- **Severity:** medium
- **Category:** bug
- **Summary:** `listEpics` fetches all epics, then queries features for each epic, then queries stories for each feature. The number of DB round-trips grows with the product of epics and features.
- **Evidence:** `internal/api/epics.go` lines 56–77 calls `Features.ListByEpicID` inside the epic loop and `Stories.ListByFeatureID` inside the feature loop.
- **Recommendation:** Add a repository method that eagerly loads the full three-level hierarchy in a constant number of queries (e.g., fetch all relevant features and stories once, then assemble in memory).
- **Rule candidate:** Endpoints that return nested collections must bound the number of queries independently of collection size.

### 4. N+1 query problem in audit list endpoint
- **Severity:** low
- **Category:** bug
- **Summary:** `listAudit` enriches every audit log entry with the current `date_source` by issuing a separate repository query per row. For large audit logs this becomes a significant performance bottleneck.
- **Evidence:** `internal/api/audit.go` lines 47–51 calls `resolveDateSource` once per audit log entry.
- **Recommendation:** Batch-fetch the relevant features and stories referenced in the audit log, then map them in memory.
- **Rule candidate:** Enriching list responses must not perform N+1 queries; batch-load related data.

### 5. Dead code in repository/metrics.go
- **Severity:** low
- **Category:** dead-code
- **Summary:** `parseDatePtr` is defined in `repository/metrics.go` but never referenced anywhere in the codebase.
- **Evidence:** `internal/repository/metrics.go` lines 175–183. `grep -rn parseDatePtr` shows only the definition.
- **Recommendation:** Remove the unused function.
- **Rule candidate:** Remove unused package-private functions after implementation; the compiler does not flag them in library packages.

### 6. Duplicated date-patch logic in stories.go and features.go
- **Severity:** low
- **Category:** simplification
- **Summary:** `patchStoryDate` and `patchFeatureDate` contain nearly identical logic for first-assignment detection, delta-day calculation, audit-log creation, and response formatting.
- **Evidence:** `internal/api/stories.go` lines 115–163 and `internal/api/features.go` lines 131–179 differ only in entity type and repository method names.
- **Recommendation:** Extract a generic `patchDate` helper that accepts the entity type, a getter for the current entity, an update function, and an audit creator.
- **Rule candidate:** Extract shared controller logic into helpers when the same pattern appears more than once.

### 7. Out-of-scope file modified (internal/importer/types.go)
- **Severity:** low
- **Category:** workflow
- **Summary:** The story explicitly lists "Import parser (already built in story-003)" as out of scope, yet `internal/importer/types.go` was modified to alias `ImportReport` from `internal/models`. While this was necessary to break an import cycle, touching an out-of-scope module without prior documentation is a workflow violation.
- **Evidence:** Story-004 out-of-scope section. `git diff` shows `internal/importer/types.go` was modified.
- **Recommendation:** When a story requires types from an out-of-scope module, either create a pre-story refactoring task or document the exception in the story Issues section before making changes.
- **Rule candidate:** If an out-of-scope module must be touched to avoid an import cycle, document the exception in the story Issues section before making changes.

### 8. Missing test for GET on /date and /feature sub-resource paths
- **Severity:** low
- **Category:** test-gap
- **Summary:** There are no tests verifying the behavior of GET requests to date-specific or feature-specific story/feature paths. This gap allowed the routing bug in Finding 1 to go undetected.
- **Evidence:** `internal/api/api_test.go` has tests for `GET /api/stories/S-1` and `PATCH /api/stories/S-1/date`, but no `GET /api/stories/S-1/date` or `GET /api/features/F-1/date` test cases.
- **Recommendation:** Add tests for GET on sub-resource paths to assert they return 405 or 404, not 400.
- **Rule candidate:** Every route pattern must have tests for unexpected HTTP methods.

## Recommended Fixes

- [x] medium — Add HTTP method guards in `handleStory`, `handleFeature`, and related dispatchers, or switch to more specific route patterns. **Fixed:** GET handlers now validate path depth and return 404 for sub-resource paths like `/date` or `/feature` instead of dispatching to PATCH handlers.
- [x] medium — Replace `fmt.Sprintf` SQL table-name interpolation in metrics queries with an allow-list or separate query functions. **Fixed:** Added `allowedMetricTables` map and `validateMetricTable` helper; `deadlineHitRate` and `scopeCreepRate` now validate table names before formatting queries.
- [x] medium — Refactor epic list/get endpoints to avoid N+1 queries by eagerly loading features and stories in bounded queries. **Fixed:** `listEpics` now fetches all features and stories once (3 queries total) and assembles in memory. `getEpic` fetches all stories once and filters by feature IDs.
- [x] low — Batch-fetch date_source values in audit list to eliminate N+1 enrichment queries. **Fixed:** `listAudit` now collects unique feature/story IDs, fetches each entity type in a batch loop, and maps results in memory instead of querying per row.
- [x] low — Remove unused `parseDatePtr` function from `repository/metrics.go`. **Fixed:** Removed dead `parseDatePtr` function and its unused `time` import.
- [x] low — Extract shared date-patch logic into a generic helper to reduce duplication between story and feature handlers. **Fixed:** Extracted `patchDateRequest` struct, `changedByOrDefault`, and `computeDatePatch` helpers into `stories.go`; both `patchStoryDate` and `patchFeatureDate` now reuse them.
- [x] low — Document the out-of-scope `importer/types.go` change in story-004 Issues (or move it to a dedicated refactoring story next time). **Fixed:** Added explanation to story-004 Issues section describing why the import cycle necessitated the change.
- [x] low — Add tests for GET requests to `/date` and `/feature` sub-resource paths to ensure proper 405/404 responses. **Fixed:** Added `TestGetStoryDate_SubResource`, `TestGetStoryFeature_SubResource`, `TestGetFeatureDate_SubResource`, `TestGetFeatureEpic_SubResource`, and `TestGetMetricsSlip_MissingID`.
