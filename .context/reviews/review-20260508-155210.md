## Findings
### Finding 1
- Severity: medium
- Category: regression
- Summary: Health and Settings navigation now render the List view instead of a neutral placeholder.
- Evidence: `frontend/src/App.svelte` routes `primary === 'list'` to `ListView`, `primary === 'roadmap' && roadmapMode === 'gantt'` to `GanttView`, and then falls through to `<ListView />` for every other primary view. Since `navItems` still includes `health` and `settings`, clicking either now shows List content with the footer still saying `View health` or `View settings`.
- Recommendation: Restore an explicit placeholder or future screen route for `health` and `settings`; keep the List fallback limited to Roadmap/List mode, not all unhandled primary views.
- Rule candidate: When adding routed screens, preserve existing navigation states with explicit fallbacks for every declared nav item.

### Finding 2
- Severity: medium
- Category: test-gap
- Summary: Core Gantt layout behavior is complex and untested.
- Evidence: Fallow reports complexity in `frontend/src/lib/ganttLayout.ts` for `buildSprints`/`sprints` (score 11), `buildBar` (score 9), `statusTone` (score 8), `buildGanttLayout` (score 8), `collectSprintNames` (score 7), `collectDatesBySprint` (score 6), and `sprintIndexForDate` (score 6). These functions encode the story's key guarantees: sprint ordering, fallback dates, synthetic epic ordering, bar placement, status colors, and today-line placement. No `ganttLayout.test.ts` or component tests were added; verification so far is `npm run build` only.
- Recommendation: Add unit tests for `buildGanttLayout` covering sprint-name/date fallback, status tone mapping, unassigned epic ordering, expansion filtering via `visibleGanttRows`, and today-line placement; then split high-complexity helpers into smaller functions where practical.
- Rule candidate: Complex pure UI layout helpers should ship with table-driven unit tests for their ordering, fallback, and classification rules before story closeout.

### Finding 3
- Severity: low
- Category: simplification
- Summary: Gantt component prop APIs duplicate common exported prop names across multiple Svelte files, keeping the Fallow duplicate-export audit failing.
- Evidence: Fallow reports duplicate exports between `GanttGrid.svelte`, `SprintHeader.svelte`, `GanttBar.svelte`, and `RowLabels.svelte` for shared prop names such as row/rows, width, rowHeight, sprints, and columnWidth. Some overlap is normal for Svelte props, but the repeated low-level layout props make the component boundary noisier and leave the static audit unresolved.
- Recommendation: Either configure/waive Fallow for Svelte component props if this is expected, or consolidate shared layout props into typed objects such as `layout`, `visibleRows`, and `barContext` so components receive fewer repeated exports.
- Rule candidate: —

### Finding 4
- Severity: low
- Category: dead-code
- Summary: Vertical scroll sync code remains after the Gantt view was changed to use the main page scrollbar.
- Evidence: `frontend/src/screens/GanttView.svelte` still declares `labelPane`, `gridPane`, `syncingScroll`, and `syncScroll`, and attaches `on:scroll` handlers to both panes. The stylesheet now sets `.labels-pane { overflow: visible; }` and `.grid-pane { overflow-x: auto; overflow-y: hidden; }`, so there is no vertical pane scroll left to synchronize. Fallow also flags `syncScroll` complexity (score 5).
- Recommendation: Remove the vertical scroll sync state and handlers, or reintroduce a deliberate internal vertical scrolling model. If horizontal-only scrolling remains the chosen UX, keep only horizontal grid overflow and let the app shell handle vertical scrolling.
- Rule candidate: Remove interaction glue when a UX change eliminates the interaction it was written to support.
