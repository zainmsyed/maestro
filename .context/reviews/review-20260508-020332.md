## Findings

### Finding 1
- **Severity:** high  
- **Category:** bug  
- **Summary:** Expand/collapse toggles in `DataTable.svelte` do not change row visibility. The `expandedEpics`/`expandedFeatures` state logic is inverted: toggling an ID adds it to the "expanded" set, but the `isExpanded` helper treats a non-empty set as a whitelist. Since the initial state is an empty set (meaning "all expanded"), adding an ID to the set keeps it expanded, and removing it returns to "all expanded." Epic and feature rows can never be collapsed.  
- **Evidence:** `DataTable.svelte` lines 43–61. `isEpicExpanded` returns `expandedEpics.size === 0 || expandedEpics.has(epicId)`. With an empty set, this is always `true`. After `toggleEpic` adds the ID, `has` is still `true`, so the row remains expanded.  
- **Recommendation:** Replace the "expanded set" model with a "collapsed set" model. `isEpicExpanded` should return `!collapsedEpics.has(epicId)`, and `toggleEpic` should add/remove from `collapsedEpics`. Same for features.  
- **Rule candidate:** State models that use an empty-collection sentinel for "all open" must track exclusions (closed items), not inclusions (open items), or the toggle operation becomes a no-op.

### Finding 2
- **Severity:** medium  
- **Category:** bug  
- **Summary:** `FilterBar.svelte` `resetFilters` omits the `type` field, so clicking "Clear filters" leaves the Type filter active while clearing all others.  
- **Evidence:** `FilterBar.svelte` line 33: `onFilterChange({ epic: '', owner: '', sprint: '', status: '' })` — missing `type: ''`.  
- **Recommendation:** Add `type: ''` to the reset object, or spread `filters` and reset all keys programmatically.  
- **Rule candidate:** Filter reset functions must enumerate every key in the filter state shape, or derive the empty state from the type definition.

### Finding 3
- **Severity:** medium  
- **Category:** bug  
- **Summary:** Story reassignment dropdown in `DataTable.svelte` never renders because `isSynthetic` is hardcoded to `false` for all story rows in `flattenRows`. The UI condition `{#if row.type === 'story' && row.isSynthetic}` is therefore always false. The `patchStoryFeature` API client, the `featureOptions` prop, and the `onReassignStory` handler are all wired but unreachable.  
- **Evidence:** `listView.ts` line 91: `isSynthetic: false` for every story row. `DataTable.svelte` line 186: `{#if row.type === 'story' && row.isSynthetic}`.  
- **Recommendation:** Either (a) mark stories as synthetic when their parent feature is synthetic (`isSynthetic: epic.is_synthetic` or `feature.is_synthetic`), or (b) remove the unreachable story reassignment UI and the `patchStoryFeature` / `featureOptions` / `onReassignStory` wiring if story reassignment was not required.  
- **Rule candidate:** UI that is gated on a boolean flag must have at least one code path that sets the flag to `true`; otherwise the conditional block is dead code.

## Completion Summary

Story-009 successfully adds three-level hierarchy rendering, date source badges, story-specific columns, inline date editing for stories, and the Type filter. Design system compliance is solid across all new and modified components.

However, the review uncovered **one high-severity bug** (non-functional expand/collapse toggles), **two medium-severity bugs** (incomplete filter reset and unreachable story reassignment UI), and **significant test gaps** in `DataTable.test.ts` and `FilterBar.test.ts`.

The expand/collapse bug is particularly impactful because it means users cannot collapse epics or features to reduce visual noise in large roadmaps. The filter reset bug is a straightforward omission. The story reassignment issue represents either a wiring bug (if reassignment was intended) or dead code (if it was not).

Tests cover the list-view helper layer well (6 passing tests in `listView.test.ts`) but the component layer is under-tested for the new story behavior. The Fallow static analysis findings are a mix of genuine dead-code hints and Svelte-specific false positives.

**All review findings resolved.**

- **High:** Expand/collapse toggle logic corrected by switching from "expanded set" to "collapsed set" (`Record<string, boolean>`). Epic and feature rows now collapse/expand correctly.
- **Medium:** `resetFilters` includes `type: ''`. Story `isSynthetic` propagated from parent epic so orphaned stories show reassignment dropdowns. `DataTable.test.ts` updated with story fixtures, story inline editing test, date source badge test, story points test, and story reassignment test.
- **Low:** `FilterBar.test.ts` uses full `FilterState` shape and tests Type filter selection/clearing. `prevRow` parameter removed from `shouldShowRow`. `ApiError` un-exported (only used internally).

**Svelte 3 reactivity note:** The root cause of the non-working expand/collapse was that `collapsedEpics`/`collapsedFeatures` were read inside helper functions (`isEpicExpanded`, `shouldShowRow`) rather than directly in the template. Svelte 3's compiler only tracks reactive dependencies for variables that appear directly in the template; variables read inside function bodies are invisible to the reactivity system. The fix: inlined all `collapsedEpics[...]` and `collapsedFeatures[...]` references directly into the template expressions so the compiler sees them as reactive dependencies and re-renders the `{#if}` blocks when they change.

**Verification:** All 25 frontend tests pass, Go backend tests pass, production build succeeds.
