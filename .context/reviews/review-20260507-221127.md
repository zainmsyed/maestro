## Findings

### Finding 1
- **Severity:** medium
- **Category:** simplification
- **Summary:** `ListView.svelte` and `DataTable.svelte` duplicate the same row/group/sort types and table semantics, creating drift risk and making future fixes easy to miss in one file.
- **Evidence:** `frontend/src/screens/ListView.svelte` defines `SortKey`, `HealthTone`, `ListRow`, and `ListGroup`; `frontend/src/components/DataTable.svelte` defines materially identical local copies. Fallow also reported duplication between these two files.
- **Recommendation:** Extract shared list-view types into a colocated module (for example `frontend/src/screens/listView.types.ts` or `frontend/src/lib/listView.ts`) and keep the grouping/sorting projection logic there as well.
- **Rule candidate:** Shared view-model types used by both a screen and its child component must live in a single module rather than being duplicated across `.svelte` files.

### Finding 2
- **Severity:** low
- **Category:** simplification
- **Summary:** `rowSortValue` and `buildGroups` are carrying too much branching logic inline, which raises maintenance risk and already contributed to reactive/debugging churn during implementation.
- **Evidence:** Verified Fallow findings: `frontend/src/screens/ListView.svelte:173` `rowSortValue` complexity 17 and `frontend/src/screens/ListView.svelte:199` `buildGroups` complexity 11.
- **Recommendation:** Replace the large switch with a keyed comparator/value map and split `buildGroups` into `sortRows`, `groupByEpic`, and `groupBySimpleField` helpers.
- **Rule candidate:** Derived table-state functions that exceed complexity thresholds should be decomposed into pure helpers by responsibility (sort, filter, group) before closeout.

### Finding 3
- **Severity:** low
- **Category:** dead-code
- **Summary:** The frontend helper modules expose a broader public API than the story actually uses.
- **Evidence:** Verified Fallow unused-export findings in `frontend/src/lib/api.ts` and `frontend/src/lib/csvExport.ts`; manual inspection confirms `apiFetch` and `buildCsv` are exported even though the story code only consumes the higher-level `api` object and `downloadCsv` helper.
- **Recommendation:** Narrow exports to the actual public surface. Keep `apiFetch`/`buildCsv` exported only if a real consumer exists or tests depend on them.
- **Rule candidate:** Helper modules should default to private functions and export only the stable surface consumed by other modules.

### Finding 4
- **Severity:** info
- **Category:** workflow
- **Summary:** Several Fallow findings are analyzer noise rather than actionable problems.
- **Evidence:** `frontend/src/stores/project.ts` imported types are used in `ProjectState`; `frontend/package.json` dev dependencies are required by Vite/Svelte/TypeScript tooling; Fallow also flagged duplicate exports across Svelte component props, which is not a meaningful issue in this codebase.
- **Recommendation:** No code change required. Add or tune Fallow ignores for Svelte prop exports and scaffold package manifests to reduce false-positive noise.

### Finding 5
- **Severity:** medium
- **Category:** test-gap

- **Evidence:** No test files cover `ListView.svelte`, `DataTable.svelte`, filtering, grouping, inline date edit, reassignment, or CSV export. The verification relied on manual runtime checks after several reactive-state bugs were discovered.
- **Recommendation:** Add lightweight frontend tests for: (a) filter application, (b) group-by behavior, (c) sort toggles, (d) inline feature date edit calling `api.patchDate`, and (e) CSV export row shaping.
