## Findings

### 1. Vite dev server port conflicts with API proxy target
- **Severity:** medium
- **Category:** bug
- **Summary:** `vite.config.ts` configures the dev server on port 9000 and simultaneously proxies `/api` to `http://localhost:9000`. When `npm run dev` is executed, Vite binds to 9000 before the Go backend can start, causing API requests to be forwarded to Vite itself rather than the backend.
- **Evidence:** `frontend/vite.config.ts` lines 11 and 15: `server.port: 9000` and `proxy['/api'].target: 'http://localhost:9000'`.
- **Recommendation:** Move the Vite dev server to a non-conflicting port (e.g. 5173 or 3000) so the proxy can reach the Go backend on its canonical port 9000.
- **Rule candidate:** Dev-server port and API proxy target must never collide; every frontend proxy config must be validated against the backend default port.

### 2. SPA fallback for root path relies on FileServer directory indexing
- **Severity:** low
- **Category:** bug
- **Summary:** In `serveSPA`, a request for `/` produces `r.URL.Path[1:]` = `""`. `dist.Open("")` returns the root directory, so `files.ServeHTTP` handles it via directory indexing rather than the explicit `index.html` fallback below. This happens to work because `http.FileServer` serves `index.html` for directories, but it is accidental and fragile.
- **Evidence:** `cmd/maestro/main.go` line 54: `if f, err := dist.Open(r.URL.Path[1:]); err == nil { ... }`.
- **Recommendation:** Explicitly handle `r.URL.Path == "/"` (or empty trimmed path) by serving `index.html` directly, bypassing the existence check.
- **Rule candidate:** SPA fallback handlers must explicitly route `/` to `index.html` rather than relying on `FileServer` directory indexing.

### 3. No unit tests for api.ts fetch wrapper
- **Severity:** low
- **Category:** test-gap
- **Summary:** The `apiFetch` helper in `frontend/src/lib/api.ts` handles request construction, header logic, JSON/FormData branching, and error message extraction, but has no unit tests. A regression in header building or error parsing would only be caught at integration time.
- **Evidence:** `frontend/src/lib/api.ts` — `apiFetch`, `buildHeaders`, `parseResponse`, and the `api` object are all untested.
- **Recommendation:** Add a lightweight vitest setup (or use the existing Node test runner) and cover: (a) `buildHeaders` with JSON body, (b) `buildHeaders` with FormData body skipping Content-Type, (c) `ApiError` message extraction from `{error: "..."}` response, (d) 204 handling.
- **Rule candidate:** —

### 4. Design-system base font size diverges from implementation
- **Severity:** info
- **Category:** workflow
- **Summary:** `.context/design/design-system.md` declares "Base size: 13px", but `frontend/src/app.css` sets `body { font-size: 16px; }` (~23 % larger). The increase was user-requested during implementation, but the design token file is now stale.
- **Evidence:** `.context/design/design-system.md` Typography section vs. `frontend/src/app.css` `body` rule.
- **Recommendation:** Update `design-system.md` Typography to reflect the agreed base size (16px) so future stories do not re-derive the wrong scale.
- **Rule candidate:** —

### 5. Fallow unused-dev-dependencies finding is a false positive
- **Severity:** info
- **Category:** workflow
- **Summary:** Fallow flagged `frontend/package.json` devDependencies as unused, but every listed package is required: `@sveltejs/vite-plugin-svelte` (Vite plugin), `svelte` (compiler/runtime), `svelte-check` (type checking), `typescript` (tsc), `vite` (build tool).
- **Evidence:** `frontend/package.json` lines 12–16.
- **Recommendation:** No code change needed. Consider configuring Fallow ignore patterns for scaffold/package files if the noise recurs.
- **Rule candidate:** —
