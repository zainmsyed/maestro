## Findings

### 1. DropZone nests interactive elements — accessibility violation
- **Severity:** medium
- **Category:** bug
- **Summary:** The outer `.drop-zone` div declares `role="button"` and `tabindex`, but contains a real `<button class="picker-button">` inside it. This creates nested interactive controls that confuse screen readers and keyboard navigation.
- **Evidence:** `frontend/src/components/DropZone.svelte` lines 57–73. The outer div is keyboard-focusable and clickable, while the inner button is also clickable and steals focus.
- **Recommendation:** Remove `role="button"` and `tabindex` from the outer div. Keep the `on:click` handler for mouse users, but let the inner button and the hidden file input provide the only actionable targets for keyboard/AT users.
- **Rule candidate:** No element with `role="button"` may contain a nested `<button>`, `<a>`, or other interactive element.

### 2. Missing document-level dragover/drop preventDefault
- **Severity:** low
- **Category:** bug
- **Summary:** If a user drops a file outside the DropZone boundary, the browser navigates away to the local file path. There is no `window` listener preventing the default drop behavior on the document.
- **Evidence:** No global `dragover` / `drop` handlers in `App.svelte`, `Onboarding.svelte`, or `DropZone.svelte`.
- **Recommendation:** Add `window.addEventListener('dragover', e => e.preventDefault())` and `window.addEventListener('drop', e => e.preventDefault())` in `App.svelte` (or `Onboarding.svelte`) so stray drops don’t navigate.
- **Rule candidate:** Any page with a drag-and-drop upload target must prevent default document-level drag/drop to avoid accidental navigation.

### 3. Backend `postImport` has no unit tests
- **Severity:** medium
- **Category:** test-gap
- **Summary:** The new `POST /api/import` handler was added to support the frontend upload, but `internal/api/api_test.go` only covers `GET /api/import/report`. The new route is untested for success, missing-file, non-CSV, and importer-error paths.
- **Evidence:** `internal/api/import.go` has `postImport`; `internal/api/api_test.go` has `TestImportReport_Endpoint`, `TestImportReport_NotFound`, and `TestImportReport_MultipleSaves` — none exercise `POST /api/import`.
- **Recommendation:** Add tests for `POST /api/import`: valid CSV multipart upload, missing `file` field, non-CSV filename rejection, empty CSV body, and importer-error propagation.
- **Rule candidate:** Every new API endpoint must have at least one success-case and one error-case test in the same PR.

### 4. Re-importing the same CSV crashes with UNIQUE constraint error
- **Severity:** medium
- **Category:** bug
- **Summary:** The importer always attempts `INSERT` for every entity. A second upload of the same CSV into an existing database fails with `UNIQUE constraint failed: epics.id`. There is no upsert, skip-existing, or database-clear path.
- **Evidence:** User visually verified this during story-007 testing; backend returns HTTP 400 with the raw SQLite error.
- **Recommendation:** Implement one of: (a) upsert existing IDs while preserving `original_end_date`, (b) skip already-existing IDs and report them in the import summary, or (c) expose an explicit "clear and re-import" option. Until then, document the limitation.
- **Rule candidate:** Import endpoints that persist entities must define idempotency behavior (upsert, skip, or wipe) and never leak raw DB constraint errors to the client.
